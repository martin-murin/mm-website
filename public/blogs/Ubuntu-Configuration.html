Installing Ubuntu 24.04 with RAID, LUKS, and LVM setup

Introduction

With a new life chapter comes a new computer. After weighing different options, I settled on yet another ThinkPad. For my operating system, I chose Ubuntu 24.04, the latest LTS release, for its compatibility. Having used Ubuntu for 6 years, I've grown to appreciate its balance of stability and features.

This guide documents my process of setting up an advanced storage configuration with:

  • RAID for improved performance and redundancy
  • LUKS encryption for data security
  • LVM for flexible storage management

Hardware Overview

My new ThinkPad specifications:

  • AMD Ryzen Pro 9 CPU
  • NVIDIA RTX 2000 Ada 8GB GPU
  • 64GB DDR5 RAM (2 x 32GB)
  • 2x 1TB M.2 PCIe Gen 4 TLC Opal SSDs

Preparation: Setting Up the Live Environment

All steps in this guide are performed from a Live Ubuntu environment running from a bootable USB drive. To prepare this environment:

  1. Create a bootable Ubuntu USB drive (I used Ubuntu 22.04 for this installation)
  2. Boot the computer from the USB drive, selecting "Try Ubuntu" option
  3. Connect to the internet to ensure package repositories are accessible
  4. Open a terminal for running the commands in this guide

For a setup this complex, having a working environment with internet access is essential, as you'll need to install additional tools like mdadm and troubleshoot issues that arise.

Understanding RAID Levels

Before diving into the implementation, let's briefly understand the RAID levels I chose:

  • RAID 0 (Striping): Data is split across multiple drives, increasing performance by distributing I/O operations but offering no redundancy. If one drive fails, all data is lost.
  • RAID 1 (Mirroring): Data is duplicated across multiple drives, providing redundancy but using only 50% of total storage capacity.

My choice of RAID levels was driven by specific needs:

  • RAID 1 for the boot partition ensures system bootability even if one drive fails
  • RAID 0 for the data partition maximizes both performance and available storage

I opted for RAID 0 for my main data partition because I work with large datasets and need the fastest possible read/write speeds. While this increases the risk of data loss (as failure of either drive would result in complete data loss), I mitigate this by maintaining a robust backup system to an external drive. The performance benefits outweigh the risks for my specific workflow.

Storage Configuration Design

I designed my partition layout with several considerations in mind:

  1. EFI System Partition: Must remain separate and non-RAIDed, as UEFI firmware cannot boot from a RAID array.
  2. Boot Partition: Placed outside LUKS encryption because the bootloader needs to be accessible before the encryption key is entered. I chose RAID 1 for this partition to ensure the system can boot even if one drive fails.
  3. Data Partition: Used RAID 0 for maximum performance and capacity, with LUKS encryption for security.
  4. LVM on Encrypted Volume: I chose to create a single logical volume rather than splitting into separate volumes for /, /home, /tmp, /var, etc. This provides flexibility to allocate space as needed across the entire system. With my workload, I often need to process large files in various locations, and having a single volume prevents running out of space in one area while having excess in another.

Step-by-Step Installation Process

Create

1. Creating Partitions

First, I created identical partition layouts on both drives:

sudo parted /dev/nvme0n1 --script -- mklabel gpt
sudo parted /dev/nvme0n1 --script -- mkpart efi fat32 1MiB 513MiB
sudo parted /dev/nvme0n1 --script -- set 1 esp on
sudo parted /dev/nvme0n1 --script -- mkpart boot 513MiB 1537MiB
sudo parted /dev/nvme0n1 --script -- mkpart data 1537MiB 100%

sudo parted /dev/nvme1n1 --script -- mklabel gpt
sudo parted /dev/nvme1n1 --script -- mkpart efi fat32 1MiB 513MiB
sudo parted /dev/nvme1n1 --script -- set 1 esp on
sudo parted /dev/nvme1n1 --script -- mkpart boot 513MiB 1537MiB
sudo parted /dev/nvme1n1 --script -- mkpart data 1537MiB 100%

Here I'm using parted to create GPT partition tables on both drives with identical layouts:

  • EFI partition (512MB): This needs to be at the beginning of the disk for compatibility with UEFI firmware
  • Boot partition (1GB): Separate from the encrypted volume to allow system boot
  • Data partition (remainder of disk): Will be used for RAID 0, encryption, and LVM

The set 1 esp on command marks the first partition as an EFI System Partition, which is necessary for UEFI boot.

2. Preparing the Partitions

To ensure clean partitions, I wiped any existing filesystem signatures:

sudo wipefs -a /dev/nvme0n1p2
sudo wipefs -a /dev/nvme1n1p2
sudo wipefs -a /dev/nvme0n1p3
sudo wipefs -a /dev/nvme1n1p3

As an extra precaution, I zeroed the first 10 MB of the boot partitions to clear out potential residual headers:

sudo dd if=/dev/zero of=/dev/nvme0n1p2 bs=1M count=10
sudo dd if=/dev/zero of=/dev/nvme1n1p2 bs=1M count=10

This step is crucial, especially when reusing drives that might have had previous filesystem signatures or RAID metadata. The wipefs command removes filesystem signatures, while the dd command overwrites the beginning of each partition with zeros, ensuring no remnants of previous configurations remain, which could confuse the installer or cause issues during RAID setup.

3. Creating EFI System Partitions

I formatted the EFI partitions with FAT32:

sudo mkfs -t vfat /dev/nvme0n1p1
sudo mkfs -t vfat /dev/nvme1n1p1

EFI System Partitions must be formatted as FAT32 because the UEFI specification requires this filesystem format. The UEFI firmware can only read FAT32 partitions, so this is a non-negotiable requirement for EFI boot to work. Note that I'm only formatting the partitions, not setting up RAID for them, as UEFI cannot boot from a RAID array.

4. Setting Up RAID Arrays

I created two RAID arrays:

# RAID 1 for boot partition (redundancy)
sudo mdadm --create /dev/md0 --level=1 --raid-devices=2 --metadata=0.90 /dev/nvme0n1p2 /dev/nvme1n1p2
# RAID 0 for data partition (performance)
sudo mdadm --create /dev/md1 --level=0 --raid-devices=2 /dev/nvme0n1p3 /dev/nvme1n1p3

For the boot partition, I specifically use metadata version 0.90 (an older format) for compatibility with bootloaders. This is critically important because metadata version 0.90 places the RAID metadata at the end of the partition rather than at the beginning. The bootloader expects to find boot metadata at the start of the partition, so using a newer metadata version (which would place RAID metadata at the beginning) would prevent the system from booting.

For the data partition, I use RAID 0 for maximum performance and the default metadata version (1.2), since this partition doesn't need to be bootable.

5. Encrypting the Data RAID Array

I encrypted the RAID 0 array with LUKS:

sudo cryptsetup luksFormat /dev/md1
sudo cryptsetup open /dev/md1 cryptraid

Here I'm using LUKS (Linux Unified Key Setup) to encrypt the entire RAID 0 array. The luksFormat command initializes the encryption, and you'll be prompted to create a strong passphrase. The open command creates a mapping at /dev/mapper/cryptraid, which is a decrypted view of the LUKS container that can be used like a regular block device.

I specifically chose to encrypt the RAID 0 array rather than individual drives before RAID because:

  1. It's simpler to manage a single encrypted volume
  2. It avoids potential performance issues with encryption happening at multiple levels
  3. It ensures the entire data partition is protected, including RAID metadata

6. Setting Up LVM

I created a logical volume on the encrypted RAID array:

sudo pvcreate /dev/mapper/cryptraid
sudo vgcreate vgubuntu /dev/mapper/cryptraid
sudo lvcreate -l 100%FREE -n root vgubuntu

These commands create a physical volume (PV), a volume group (VG) named "vgubuntu", and a logical volume (LV) named "root" that uses all available space. LVM provides flexibility for future storage management. If I later need to resize partitions or add more storage, I can do so without repartitioning the entire disk.

I chose to create a single logical volume rather than separate ones for different mount points because I need the flexibility to use space wherever it's needed in my system.

7. Formatting Filesystems

I formatted the boot partition and LVM root volume:

sudo mkfs.ext4 /dev/md0
sudo mkfs.ext4 /dev/vgubuntu/root

8. Ubuntu Installation

I encountered a significant hurdle at this point: Ubuntu 24.04's graphical installer has removed the "Something else" option that was present in previous versions. This makes it impossible to install Ubuntu 24.04 directly using the graphical installer with this advanced storage configuration.

After exploring my options, I had three potential paths forward:

  1. Use debootstrap to perform a manual installation
  2. Use the Ubuntu Server installer and then manually install the Ubuntu Desktop packages
  3. Use the Ubuntu 22.04 installer and upgrade to 24.04 afterward

I opted for the third option as it was the most familiar to me. The Ubuntu 22.04 installer still includes the "Something else" option that allows manual partition assignment.

Important: When running the graphical installer, ensure that the LUKS volume is already open and decrypted (cryptraid should be available). Otherwise, the installer won't be able to see the logical volume with the target partition. If Live Linux was rebooted, use the following commands:

sudo apt update
sudo apt install mdadm
sudo mdadm --assemble --scan
sudo cryptsetup open /dev/md1 cryptraid

In the installer, I selected "Something else" for partitioning and specified:

  • /dev/vgubuntu/root as the mount point for /
  • /dev/md0 as the mount point for /boot
  • Selected /dev/nvme0n1p1 as the bootloader location

While the operating system installed successfully, the bootloader installation failed. This could be due to the complexity of the RAID setup or because the OS was installed into an LVM volume under RAID. Either way, manual intervention was required to get the system bootable.

Troubleshooting the Installation

The graphical installer failed to configure GRUB correctly, so I had to perform manual configuration:

1. Understanding Chroot

Chroot (Change Root) is a powerful operation that changes the apparent root directory for the current running process and its children. By doing chroot into a directory like /mnt, that directory becomes the new root (/) from the perspective of processes inside the chroot environment. This allowed me to work within a separate Linux installation as if I had booted directly into it.

2. Mounting Filesystems for chroot

sudo mount /dev/vgubuntu/root /mnt
sudo mount /dev/md0 /mnt/boot
sudo mount /dev/nvme0n1p1 /mnt/boot/efi

# Mount special filesystems
sudo mount -t proc none /mnt/proc
sudo mount --rbind /sys /mnt/sys
sudo mount --rbind /dev /mnt/dev
sudo mount --rbind /run /mnt/run
sudo mount -t devpts none /mnt/dev/pts

The chroot environment requires these mounts to function properly. The special filesystems (proc, sys, dev, run, devpts) are necessary because they provide essential system information and functionality:

  • proc: A virtual filesystem providing process and kernel information, needed by many system utilities
  • sys: Exposes kernel parameters and device information to userspace
  • dev: Contains device files that programs use to interact with hardware
  • run: Contains runtime variable data like PID files and sockets
  • devpts: A pseudoterminal filesystem that's absolutely critical for interactive commands like text editors and shells to work properly inside the chroot environment - without it, many interactive programs will fail with terminal-related errors

The --rbind option ensures that any existing mounts within these directories are also carried over to the chroot environment, maintaining the complete hierarchy.

3. Chrooting and Installing Required Packages

sudo chroot /mnt /bin/bash
apt update
apt install lvm2 cryptsetup mdadm

Inside the chroot, I installed the essential packages needed for the system to recognize and mount the RAID arrays, decrypt the LUKS container, and access the LVM volumes during boot. Without these packages, the initramfs wouldn't include the necessary tools to mount the root filesystem.

4. Configuring System Files

I verified and updated these critical configuration files:

/etc/crypttab

cryptraid UUID=<LUKS-UUID> none luks

This file tells the system which encrypted volumes to unlock during boot. The none parameter means the system will prompt for a passphrase rather than using a keyfile.

/etc/mdadm/mdadm.conf

ARRAY /dev/md0 metadata=0.90 UUID=<uuid>
ARRAY /dev/md/1 metadata=1.2 UUID=<uuid>

This configuration ensures the RAID arrays are properly assembled during boot. The UUID values uniquely identify each array.

/etc/modules

raid1
raid0

These module entries ensure the necessary RAID kernel modules are loaded early in the boot process.

5. Updating Initramfs and GRUB

update-initramfs -u -k all
grub-install --target=x86_64-efi --efi-directory=/boot/efi --bootloader-id=ubuntu
update-grub

The update-initramfs command rebuilds the initial ramdisk with the necessary modules and configuration to mount the RAID arrays, decrypt the LUKS container, and access the LVM volumes. The -k all option ensures this is done for all installed kernels.

The grub-install command installs the GRUB bootloader to the EFI partition. The --target=x86_64-efi option specifies that this is an EFI system, and --efi-directory=/boot/efi tells GRUB where the EFI partition is mounted.

Finally, update-grub generates the GRUB configuration file, which includes entries for the installed kernels and other detected operating systems.

6. Exiting Chroot and Rebooting

Once all configurations were complete, I exited the chroot environment:

exit

Then, I attempted to unmount the mounted filesystems:

sudo umount -l /mnt/dev/pts
sudo umount -l /mnt/proc
sudo umount -l /mnt/sys
sudo umount -l /mnt/dev
sudo umount -l /mnt/run
sudo umount -l /mnt/boot/efi
sudo umount -l /mnt/boot
sudo umount -l /mnt

However, I consistently encountered "filesystem busy" errors when trying to unmount them properly. Instead of troubleshooting further, I simply rebooted the live Linux environment:

sudo reboot

After rebooting, I booted into my newly installed Ubuntu 22.04.

7. Upgrading Ubuntu

When upgrading Ubuntu, I encountered issues with the 1GB boot partition size. To resolve this:

  1. I removed old kernels:
    dpkg --list | grep linux-image
    sudo apt-get purge linux-image-x.x.x-x-generic
  2. I set compression for initramfs in /etc/initramfs-tools/initramfs-config:
    COMPRESS=xz
  3. Then I could successfully run:
    do-release-upgrade

Post-Installation Considerations

RAID Maintenance

I configured RAID maintenance options:

dpkg-reconfigure mdadm

For my RAID 1 boot partition (/dev/md0), I opted for monthly redundancy checks (last Sunday of the month) and disabled daily checks and email notifications. This ensures the mirrored data stays consistent over time.

For the RAID 0 data partition (/dev/md/1), redundancy checks aren't applicable since RAID 0 has no data redundancy - it simply stripes data across disks for improved performance. With RAID 0, regular backups are essential since the failure of any single drive will result in complete data loss.

Conclusion

Setting up Ubuntu 24.04 with RAID, LUKS encryption, and LVM was definitely a deep dive, but the end result is worth it. This setup gives me exactly what I need: strong security, solid performance, and the flexibility to adapt as my storage needs grow.

RAID 0 keeps my data fast and accessible. LUKS encryption adds a crucial layer of security, and LVM lets me manage storage with ease. Sure, the manual configuration took some effort, but now I have a system that works my way, optimized for performance and reliability.
